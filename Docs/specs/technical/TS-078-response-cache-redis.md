# TS-078 — Response Caching Layer (Upstash Redis, 24h TTL, keyed on prompt hash)

**Author:** CoFounder (Agent 1)
**Date:** 2026-04-07
**Priority:** P1 — secondary cost lever, not a blocker for pricing
**Scope:** GEO app (llm-gateway + citation-checker)
**Depends on:** TS-077 (requires the gateway abstraction, the `rephraseSeeds` determinism fix, and the padded prompt constants)

---

## 1. What

Add a response-level cache layer in front of the LLM gateway, backed by Upstash Redis, keyed on a deterministic hash of `(model_id, system_prompt_version, user_prompt, evidence_hash)`. Cache hits skip the LLM call entirely and return the previously-computed response. Cache entries have a 24-hour TTL.

The cache lives **inside** `llm-gateway.ts` (see TS-077 §5.1), transparently to every call site. Call sites do not need to change — they just suddenly pay zero on repeat queries within 24 hours.

## 2. Why

### 2.1 Real users send the same prompts more than once

Observed patterns from existing citation-check logs:

- 2-checks-per-day usage model (Aditya's projection from the cost audit): the same site runs the same citation checks twice a day. Same site → same category tree → same real-prompt discovery → same 44 prompts per run → same evidence from Brave → **same final responses**.
- Across different customers: high-overlap prompts for common verticals ("best CRM tools", "alternatives to Slack", "top project management software") — these are generated by `citation-prompt-generator.ts` deterministically from category trees, so two dental clinics in the same city generate overlapping prompts.

At a 50% cache hit rate on the preemptive-injection warm path ($1.06/check), response caching drops the amortized cost to **~$0.58/check**. At 80% hit rate (repeat customers), amortized cost drops to **~$0.35/check**.

### 2.2 The cost lever compounds with prompt caching

TS-077's prompt caching knocks per-call input cost down by 90% on Anthropic. Response caching skips the call entirely on cache hit. These are orthogonal — prompt caching helps miss-path cost, response caching eliminates calls outright. Both should be on.

### 2.3 Prompt determinism is already being fixed in TS-077

TS-077 fixes `rephraseSeeds` temperature to 0, which is the prerequisite for any exact-match cache to work. With that fix in place, TS-078 is a self-contained, low-risk addition.

## 3. Scope

### 3.1 In scope

- New file: `lib/services/llm-response-cache.ts` — Redis get/set/hash helpers
- Modification: `lib/services/llm-gateway.ts` — wrap `measurementCall()` and `auxiliaryCall()` with cache lookup
- New env var: `RESPONSE_CACHE_ENABLED` (boolean flag, default `true` in production, `false` in staging initially for shadow testing)
- New env var: `RESPONSE_CACHE_TTL_HOURS` (default `24`)
- New columns on `llm_call_log` (from TS-077): `cache_hit` boolean, `cache_key` text
- New metric: cache hit rate surfaced to `/api/internal/metrics/llm-cache`

### 3.2 Out of scope

- Semantic / fuzzy cache (e.g., vector similarity). Exact-match only in v1.
- Cache warming on scheduled cron. Real user traffic populates the cache.
- Cross-region replication. Upstash Redis global database handles this at the provider layer.
- Cache invalidation on model version change — instead, we include the `model_id` in the cache key, so a model change naturally bypasses stale entries.

## 4. Cache key design

```
cache_key = sha256(
  model_id + "|"
  + system_prompt_version + "|"
  + normalize(user_prompt) + "|"
  + evidence_hash
)
```

Where:

- `model_id` — full OpenRouter model string (e.g., `anthropic/claude-haiku-4-5`). Prevents stale hits when we change models.
- `system_prompt_version` — a manually-bumped string constant in `lib/services/prompts/citation-system-prompt-{provider}.ts`, e.g., `"v1-2026-04-07"`. When we edit a system prompt, bump the version and all cached entries for that provider expire naturally.
- `normalize(user_prompt)` — lowercase, collapse whitespace, trim. Makes "best CRM" and "Best  CRM " share a cache entry.
- `evidence_hash` — sha256 of the **ordered** Brave result list, specifically `title + url + description` for each result. Captures the evidence state. Two calls with the same user prompt but different Brave results (because Brave's index changed between them) get different cache keys.

Final cache key is prefixed with a namespace for easy flushing:

```
redis_key = "llm:resp:v1:" + sha256_hex
```

The `v1` segment lets us flush the entire cache by changing the namespace, without needing to enumerate keys.

## 5. Interfaces

### 5.1 `lib/services/llm-response-cache.ts`

```typescript
import { Redis } from "@upstash/redis";

// Idiomatic init — reads UPSTASH_REDIS_REST_URL and UPSTASH_REDIS_REST_TOKEN
// automatically from process.env. Provisioned via Vercel Marketplace
// integration (`vercel integration add upstash`), already present in platform stack.
const redis = Redis.fromEnv();

const NAMESPACE = "llm:resp:v1:";
const TTL_SECONDS = Number(process.env.RESPONSE_CACHE_TTL_HOURS ?? "24") * 3600;

export interface CacheEntry {
  text: string;
  modelId: string;
  cachedAt: string;        // ISO-8601
  upstreamProvider: string;
  inputTokens: number;
  outputTokens: number;
}

export function computeCacheKey(input: {
  modelId: string;
  systemPromptVersion: string;
  userPrompt: string;
  evidenceHash: string;
}): string {
  const normalized = input.userPrompt.toLowerCase().replace(/\s+/g, " ").trim();
  const raw = `${input.modelId}|${input.systemPromptVersion}|${normalized}|${input.evidenceHash}`;
  return NAMESPACE + sha256Hex(raw);
}

export function computeEvidenceHash(evidence: BraveSearchResult[]): string {
  const serialized = evidence
    .map(r => `${r.title}||${r.url}||${r.description}`)
    .join("\n");
  return sha256Hex(serialized);
}

export async function getCachedResponse(key: string): Promise<CacheEntry | null> {
  if (process.env.RESPONSE_CACHE_ENABLED !== "true") return null;
  try {
    const entry = await redis.get<CacheEntry>(key);
    return entry ?? null;
  } catch (err) {
    // Redis outage: fail open, miss the cache, log it
    console.warn("[llm-cache] redis get failed", err);
    return null;
  }
}

export async function setCachedResponse(key: string, entry: CacheEntry): Promise<void> {
  if (process.env.RESPONSE_CACHE_ENABLED !== "true") return;
  try {
    await redis.set(key, entry, { ex: TTL_SECONDS });
  } catch (err) {
    // Redis outage: fail open, the next call will miss and re-query
    console.warn("[llm-cache] redis set failed", err);
  }
}

function sha256Hex(input: string): string {
  return crypto.createHash("sha256").update(input).digest("hex");
}
```

### 5.2 Integration inside `llm-gateway.ts`

`measurementCall()` (pseudo-code; extends the TS-077 signature):

```typescript
export async function measurementCall(input: MeasurementCallInput): Promise<MeasurementCallOutput> {
  const evidenceHash = computeEvidenceHash(input.evidence);
  const systemPromptVersion = SYSTEM_PROMPT_VERSIONS[input.provider]; // "v1-2026-04-07"
  const cacheKey = computeCacheKey({
    modelId: MEASUREMENT_MODELS[input.provider],
    systemPromptVersion,
    userPrompt: input.prompt,
    evidenceHash,
  });

  // CACHE HIT PATH
  const cached = await getCachedResponse(cacheKey);
  if (cached) {
    await logLlmCall({
      taskKey: `citation-check.${input.provider}`,
      modelId: cached.modelId,
      upstreamProvider: cached.upstreamProvider,
      inputTokens: cached.inputTokens,
      cachedInputTokens: 0,    // provider-level cache is orthogonal
      outputTokens: cached.outputTokens,
      costUsd: 0,              // zero cost on cache hit
      latencyMs: 0,            // negligible, we don't measure Redis round-trip here
      cacheHit: true,
      cacheKey,
    });
    return {
      text: cached.text,
      inputTokens: cached.inputTokens,
      outputTokens: cached.outputTokens,
      cachedInputTokens: 0,
      costUsd: 0,
      latencyMs: 0,
      upstreamProvider: cached.upstreamProvider,
      modelId: cached.modelId,
    };
  }

  // CACHE MISS PATH — execute the real call
  const result = await executeMeasurementCall(input);

  // Populate cache (fire-and-forget; don't block response on Redis write)
  void setCachedResponse(cacheKey, {
    text: result.text,
    modelId: result.modelId,
    cachedAt: new Date().toISOString(),
    upstreamProvider: result.upstreamProvider,
    inputTokens: result.inputTokens,
    outputTokens: result.outputTokens,
  });

  await logLlmCall({
    taskKey: `citation-check.${input.provider}`,
    modelId: result.modelId,
    upstreamProvider: result.upstreamProvider,
    inputTokens: result.inputTokens,
    cachedInputTokens: result.cachedInputTokens,
    outputTokens: result.outputTokens,
    costUsd: result.costUsd,
    latencyMs: result.latencyMs,
    cacheHit: false,
    cacheKey,
  });

  return result;
}
```

`auxiliaryCall()` gets the same treatment but with a different namespace (`llm:aux:v1:`) so measurement and auxiliary caches never collide.

## 6. Data model changes

Two additive columns on `llm_call_log` (introduced in TS-077):

```sql
-- Migration: T225-llm-cache-columns
ALTER TABLE llm_call_log
  ADD COLUMN cache_hit boolean NOT NULL DEFAULT false,
  ADD COLUMN cache_key text;

CREATE INDEX idx_llm_call_log_cache_hit
  ON llm_call_log (cache_hit, created_at DESC)
  WHERE cache_hit = true;
```

The partial index makes cache-hit-rate queries fast without bloating the index for the miss path.

## 7. Observability

Surface cache hit rate via a new internal metrics endpoint:

```typescript
// app/api/internal/metrics/llm-cache/route.ts
export async function GET(request: Request) {
  // Auth: internal-only, check for valid session cookie + admin role
  const { hits, misses } = await db.one(`
    SELECT
      SUM(CASE WHEN cache_hit THEN 1 ELSE 0 END)::int AS hits,
      SUM(CASE WHEN NOT cache_hit THEN 1 ELSE 0 END)::int AS misses
    FROM llm_call_log
    WHERE created_at > now() - interval '24 hours'
  `);
  const total = hits + misses;
  return Response.json({
    window: "24h",
    totalCalls: total,
    hits,
    misses,
    hitRate: total > 0 ? hits / total : 0,
  });
}
```

A dashboard widget consuming this endpoint can live in `app/(admin)/cost-dashboard/` (separate UI spec later, not in scope for this TS).

## 8. Rollout plan

### 8.1 Phase 1 — Ship cache layer with flag off

1. Ship `llm-response-cache.ts`, cache key logic, gateway wrapping
2. Deploy with `RESPONSE_CACHE_ENABLED=false` in production
3. Verify gateway still functions (miss path == full LLM call, no behavior change)

### 8.2 Phase 2 — Enable for auxiliary calls only

1. Set `RESPONSE_CACHE_ENABLED=true` in staging, run shadow audits
2. Measure cache hit rate on auxiliary calls (`tree-extraction`, `category-extraction`, etc.) — these should have high hit rates because inputs are derived from scraped content which is stable per site
3. Expected hit rate: ≥ 60% within 24h of steady operation
4. Flip flag on in production **for auxiliary calls only** — leave measurement calls going through the miss path via a per-call `skipCache: true` override

### 8.3 Phase 3 — Enable for measurement calls

1. Validate that `rephraseSeeds` temperature fix is actually deterministic in production (check 5 recent replays hash-identically)
2. Remove the `skipCache: true` override on measurement calls
3. Monitor for cache hit quality issues: are users getting stale answers? Track via a sampled comparison job that re-runs 5% of cached responses and diffs against a fresh call
4. Expected hit rate on measurement calls: 30–50% in the first week, climbing to 60–80% as the cache warms with repeat users

### 8.4 Phase 4 — Dashboard + alerting

1. Ship `/api/internal/metrics/llm-cache` endpoint
2. Build minimal admin widget showing cache hit rate, cost saved vs. miss path
3. Slack alert on hit rate drops below 40% (may indicate non-determinism regression)

## 9. Acceptance criteria

### 9.1 Functional

- ✅ Cache hit produces byte-identical response to the original LLM call that populated it
- ✅ Cache miss executes the real LLM call and populates the cache for next time
- ✅ Redis outage does NOT break citation checks — fail open to the LLM call
- ✅ TTL expiry removes stale entries (verify by waiting 25h after writing, confirming the next call misses)
- ✅ Model change (e.g., `claude-haiku-4-5` → `claude-haiku-4-6`) naturally bypasses the cache because `model_id` is in the key
- ✅ System prompt version bump (`v1-2026-04-07` → `v2-2026-05-01`) naturally bypasses the cache

### 9.2 Economic

- ✅ Cache hit rate on auxiliary calls ≥ 60% within 48h of enabling
- ✅ Cache hit rate on measurement calls ≥ 30% in week 1, ≥ 50% in week 4
- ✅ Amortized cost per citation check ≤ $0.70 at 30% hit rate, ≤ $0.58 at 50%, ≤ $0.35 at 80%

### 9.3 Operational

- ✅ Cache get/set latency P95 ≤ 50ms (Upstash Redis global)
- ✅ No citation check regressions attributable to cache reads
- ✅ Per-day cache storage on Upstash stays under 100MB (TTL-capped, text-only entries)

## 10. Risks

### 10.1 Non-determinism leaks back in

**Risk:** Even after the `rephraseSeeds` fix, some call sites pass non-deterministic inputs (random seeds, timestamps in prompts, user-specific data that varies per call).

**Likelihood:** Medium. The codebase is large and there may be other temperature > 0 or injected timestamp patterns we haven't found.

**Mitigation:** The cache key captures the *full* user prompt, so any timestamp or varying content lands in the hash and produces a miss. This is actually correct behavior — if the input varies, the cache should miss. The risk is only on **false** non-determinism: inputs that look identical but aren't (e.g., trailing whitespace). The `normalize()` function handles whitespace. Monitor for suspiciously low hit rates and investigate.

### 10.2 Stale cache serving wrong answers

**Risk:** A cached answer from 23 hours ago references a brand that has since gone out of business or rebranded. User gets an outdated citation.

**Likelihood:** Real but rare at 24h TTL.

**Mitigation:**
1. 24h TTL is a deliberate bound — longer TTL = more stale risk, shorter TTL = lower hit rate. 24h is the compromise.
2. Surface "data freshness" timestamp in the UI (which we already do — citation check shows last-refresh time).
3. Users can force-refresh a check, which can bypass the cache via a `skipCache: true` param on the API route — this is a product decision for TS-079's UI updates, not this spec.

### 10.3 Cache key collisions

**Risk:** sha256 collision produces two distinct inputs mapping to the same cache key.

**Likelihood:** 1 in 2^256. Not a real concern.

**Mitigation:** None needed.

### 10.4 Upstash cost blows up

**Risk:** Per-request Upstash pricing combined with high traffic makes Redis a line item.

**Likelihood:** Low. Upstash Redis Pay-as-you-go: $0.20 per 100k requests. At 100 customers × 200 checks/mo × 44 prompts × 4 providers = 3.5M gateway calls/mo = $7/mo Upstash cost. Negligible.

**Mitigation:** None needed. Monitor via Upstash dashboard.

### 10.5 Cache miss storms on cold deploys

**Risk:** When we bump the system prompt version, all caches invalidate simultaneously, causing a cost spike on the miss path.

**Likelihood:** Certain on every system prompt bump.

**Mitigation:**
1. Only bump versions when the system prompt change is intentional and impactful
2. Schedule bumps during off-peak hours
3. Document the cost impact of a version bump in the prompt-change commit message

## 11. Open questions for Aditya

1. **Cache TTL:** 24 hours is the default. Shorter (12h) is more fresh but cheaper cache. Longer (72h) is more hit-rate but more stale risk. Default 24h; confirm.
2. **Force-refresh UX:** Should the UI expose a "force refresh" button that bypasses the cache? Default: yes, but ship it in TS-079 (pricing/UX) not this spec.
3. **Cache flush mechanism:** How do we invalidate the entire cache in an emergency? Default: bump the `v1` namespace in `llm-response-cache.ts` to `v2`, deploy. All old keys become unreachable and TTL-expire naturally over the next 24h. Acceptable?
