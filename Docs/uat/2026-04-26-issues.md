## B. API routes / pipeline behavior

| # | Sev | Item |
|---|---|---|
| B1 | 🟠 | `pipeline_status` is **not updated to `failed`** when `extract-trees` permanently fails. Walmart's first audit failed (Anthropic 401), credits refunded, but DB row stayed `pending`. Dashboard rendered it as "Discovering pages…" indefinitely (zombie-pending). Stage failure handler at `app/api/pipeline/stage/route.ts` must always `UPDATE geo_sites SET pipeline_status='failed' …` before exiting. |
| B2 | 🟠 | `POST /api/auth/otp/send` 500 path returns **empty body** on rate-limit DB insert failure. Browser sees `Unexpected end of JSON input`. Other 500 paths correctly return `{ error: ... }`. Audit all catch blocks in `app/api/auth/otp/send/route.ts` (and similar routes) to JSON-encode every 5xx exit. |
| B3 | 🟠 | `POST /api/sites` re-audit flow for an authenticated Pro user gates on `email_verified` and **silently does not restart the pipeline** if user is unverified. Two consecutive submits at 02:22 + 02:24 only sent OTP emails; pipeline never restarted. Pro session should auto-pass the email-verified gate (we're already authenticated), or UI should make the OTP step explicit. |
| B4 | 🟠 | `POST /api/sites/[id]/regenerate` returns **HTTP 200**, but `RowActions.tsx:57` only triggers `onScanStart?.()` on `res.status === 202`. Mismatched protocol → optimistic-scan state never set → polling never starts → UI doesn't update until manual refresh. Either route returns 202 on success or client accepts 200. |
| B5 | 🟡 | `selectTopUrlsWithGemini` in `lib/services/geo-crawler.ts:623` swallows errors silently — `catch { return null; }` with **no logging**. For walmart (83,000 URLs) the prompt almost certainly exceeded Gemini 2.5 Flash's 1M-token input limit (5–7MB of URL text). Falls back to deterministic priority-sort silently. Add `console.warn(err)` to the catch; consider URL-list chunking for sites with >50k URLs. |
| B6 | 🟡 | `[content-generator] LLM response failed schema validation: SyntaxError: Unterminated string in JSON` fires repeatedly during `generate-chunk` stage (4+ occurrences during walmart audit at positions 12215, 12614, 12666, 14732). Likely max_tokens cutoff before LLM finishes JSON. System retries / falls back, but no metric / no operator visibility. |

## D. Infra / docker / config

| # | Sev | Item |
|---|---|---|
| D1 | 🔴 | **Vercel-stored env values have literal `\n` corruption.** `DATABASE_URL` and `NEXT_PUBLIC_WEBSITE_URL` both end with `\n` inside their quoted values — when env parsers interpret quoted strings, `\n` becomes a real newline, making the URL invalid (`postgres<newline>` → `database "postgres\n" does not exist`). Likely from `echo "…" \| vercel env add` with trailing newline. Strip and re-add via `vercel env rm` + `vercel env add` (Option B path Aditya deferred). |
| D2 | 🟠 | `.env.docker` had **stale credentials** vs prod — DATABASE_URL password rotated upstream sometime after 2026-04-10, Anthropic and 16 other API keys also stale. Local UAT silently 401'd / 28P01'd until refresh. No automated sync mechanism. Establish a `npm run env:refresh-docker` (wraps `vercel env pull` + key-merge) or document the manual flow. |
| D3 | 🟠 | `.dockerignore` missing critical entries — `playwright-report`, `test-results`, `e2e-results`, `e2e/.playwright-storage-state.json`, `.docker-test`, `.parked-tests`, `coverage`, `.vercel`, `supabase/.branches`, `supabase/.temp`, `.env.local.bak.*`, `.env.docker.bak.*`, `.env.vercel-prod`. Build fails on host symlinks (e.g. `playwright-report → /home/aditya/data/flowblinq-artifacts/...`). Fix is currently a worktree mod on the laptop, **uncommitted**. Land as part of e2e-comprehensive-suite. |
| D4 | 🟡 | `docker compose restart` does **not reload env_file** — env vars are baked at container CREATE time. After patching `.env.docker`, you need `docker compose down && up`, not `restart`. Even `--force-recreate` was unreliable in this session — full down+up was required. Document in docker UAT runbook. |
| D5 | 🟡 | Cloudflared tunnel URL drift. Each `cloudflared tunnel --url localhost:PORT` produces a random `*.trycloudflare.com` URL. `.env.docker` had `pos-helena-…` (defunct); current tunnel is `asian-edgar-…`. Set up a **named tunnel** with stable URL, or auto-update `.env.docker` from the running cloudflared process. |
| D6 | 🟡 | docker-compose env_file path resolution gotcha — paths resolve relative to **project dir** (first `-f` file's parent), NOT to the override file declaring them. `env_file: ../.env.docker` in `.docker-test/compose.override.yml` resolved to `/home/aditya/flowblinq/.env.docker` (parent of `geo/`), not to repo root. Use `.env.docker` (relative to project dir) in overrides. Document. |
| D7 | 🟡 | NEXT_PUBLIC_* values are **exposed to client JS** by Next.js — when set to `localhost:3030` and the app is accessed from another machine on LAN (`192.168.1.10:3030`), the **browser** sees `localhost` and tries to fetch its own loopback. Form submissions silently fail. For LAN-accessible UAT, NEXT_PUBLIC_APP_URL must be the LAN IP, not localhost. |

## G. Post-audit action issues (added 2026-04-26 02:50 IST)

| # | Sev | Item |
|---|---|---|
| G1 | 🟠 | **Access-token rotation invalidates open audit pages**. `POST /api/sites/[id]/regenerate` rotates the accessToken on every re-run (`buildRegeneratePatch()` returns `nanoid(32)` for `accessToken`). If the user has the audit page (`/sites/[id]`) open in another tab from before the rotation, every subsequent action button (Map Competitors, Add Competitor, Rerun Citations, Download ZIP, Download PDF) sends the **stale** token from the pages server-rendered `accessToken` prop and gets **401 Unauthorized** from the server. Fix options: (a) trigger a `router.refresh()` after regenerate completes so the new token is pulled into client props, (b) accept both old and new tokens during a grace window, (c) use Supabase JWT instead of per-site rotated tokens for action authorization. |
| G2 | 🟠 | **Map Competitors silently fails on the client.** `app/sites/[id]/SitePageClient.tsx:374` does `if (!res.ok || !res.body) { setCompetitorScanActive(false); return; }` — no error toast, no console log, no user-visible feedback. User sees the loading spinner clear and nothing else. Same anti-pattern as B5 (silent Gemini failure). Surface a toast or inline error message with the actual server error. |
| G3 | 🔴 | **`SitePageClient.tsx` caches access token in `sessionStorage` and never refreshes it.** At `app/sites/[id]/SitePageClient.tsx:194`, the token bootstrap logic checks `sessionStorage.getItem(geo-token-${siteId})` FIRST. If anything is stored, it uses that and ignores the freshly server-rendered `initialSite.token`. Once a token is cached, subsequent page loads (including hard refresh) keep using the stale token — even after a token rotation via regenerate. The audit pages own Rerun handler (lines 314–327) correctly updates sessionStorage post-regenerate, but the dashboards Rerun (RowActions.tsx) cannot — its on a different route with no access to the audit pages sessionStorage. So **dashboard-initiated re-audits leave the audit pages token stale indefinitely**. All action buttons (Map Competitors, Add Competitor, Rerun Citations, Download ZIP/PDF) then 401 silently. Fix: invert the bootstrap to prefer `initialSite.token` over stored, or detect mismatch and overwrite. |
